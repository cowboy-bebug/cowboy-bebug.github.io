Beware the Enum Gotcha in React Conditional Rendering

Posted at 2025-04-04 # bug # react # typescript

Recently at work, I ran into a sneaky bug while working with a React component and a TypeScript enum. If you’ve ever used enums to drive UI behavior (like rendering an icon or status), this one might trip you up too.

The Setup

Imagine you’re building a button component that optionally displays an icon based on a TypeScript enum:

export enum Status {
  Success,
  Warning,
  Error,
}

And you have a helper to map the enum to an icon:

const getStatusIcon = (status: Status): string => {
  switch (icon) {
    case Status.Success:
      return "✔️";
    case Status.Warning:
      return "⚠️";
    case Status.Error:
      return "❌";
  }
};

Now, you use this inside your React component:

const StatusButton: React.FC<{ label: string; status?: Status }> = (props) => {
  const icon = props.status ? getStatusIcon(props.status) : null;
  return (
    <button>
      {icon && <span>{icon}</span>} {props.label}
    </button>
  );
};

Looks fine, right? But then you notice that when status is Status.Success, no icon renders.

The Bug

This happens because Status.Success equals 0. In JavaScript, 0 is falsy, so the condition props.status ? fails even though the value is valid.

TL;DR:

props.status ? ... // ❌ Skips when status === 0

The Fix ✅

Replace the falsy check with a strict undefined check:

const StatusButton: React.FC<{ label: string; status?: Status }> = (props) => {
  const icon = props.status !== undefined ? getStatusIcon(props.status) : null;
  return (
    <button>
      {icon && <span>{icon}</span>} {props.label}
    </button>
  );
};

Now even if status === 0 (or Status.Success), it’ll still work properly.

Takeaway

When using enums in JSX or conditional rendering:

  • Avoid implicit boolean checks (if (enumValue))
  • Use !== undefined to check if an enum prop is present

It’s a small tweak that can save hours of head-scratching when something mysteriously doesn’t render.
